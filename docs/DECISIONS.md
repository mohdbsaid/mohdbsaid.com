# Architectural Decisions

Log of non-obvious decisions made during the foundation pass, and known issues discovered along the way. New entries go at the bottom. See `CLAUDE.md` for when a decision must be recorded here before a change is made.

---

### ADR-001: Adopt Content Layer collections for future content, with zero collections defined yet

**Context:** Nav links to `/blog`, `/projects`, `/courses`, `/resources` exist but no content or pages do.
**Decision:** Add `src/content.config.ts` with `export const collections = {}` and a `src/content/` folder, both documented, but define no actual collections or schemas yet.
**Why:** Gives the content model a correct home (Astro 5+ Content Layer API, not the legacy `src/content/config.ts` location) without inventing schemas for content that doesn't exist, and without creating any pages.

### ADR-002: Defer `@astrojs/rss` installation until the blog collection has real content

**Context:** Task asked for "RSS preparation."
**Decision:** Document the RSS plan in `CONTENT_STRATEGY.md` rather than installing `@astrojs/rss` or creating `src/pages/rss.xml.js` now.
**Why:** An RSS feed needs a page (`src/pages/rss.xml.js`), which the task explicitly said not to create in this pass. Installing an unused dependency ahead of that would violate "don't add things beyond what's needed now."

### ADR-003: Add `@astrojs/sitemap` now (not deferred)

**Context:** Task explicitly asked for sitemap configuration.
**Decision:** Install `@astrojs/sitemap`, set `site` in `astro.config.mjs` to a placeholder domain, add the integration.
**Why:** Unlike RSS, the sitemap integration requires no new page — it hooks into the existing build and needs no content to be useful once real pages exist.

### ADR-004: Pre-existing broken CSS in `src/styles/global.css` — left unfixed in this pass

**Context:** While configuring Prettier, `prettier --check` crashed entirely on `src/styles/global.css` (`UnexpectedNodeError: Unexpected PostCSS node type`). Investigation confirmed the file has genuinely invalid CSS: a `:root` selector (line ~24) with no opening brace, five `@font-face` blocks stranded inside `@media (prefers-color-scheme: dark)`, and a dangling `{ --color-background: #000000; ... }` block disconnected from any selector. Running `npm run build` confirms this **breaks the production build outright** (`lightningcss` throws `Invalid dangling combinator in selector`); `npm run dev` still works because Vite's dev server doesn't run the same minifier.
**Decision:** Do not fix it in this pass. Exclude the file from Prettier (`.prettierignore`) so tooling setup doesn't crash, and document it here + in `ARCHITECTURE.md` / `ROADMAP.md` as the first task of Phase 1.
**Why:** The task explicitly scoped this pass to foundation/tooling/docs with "no visual or behavioral changes." Fixing broken CSS is a legitimate, separate bug-fix task — it should land as its own change with its own review, not buried inside a structural/tooling commit. **This is flagged as a launch blocker** — the site cannot currently be built for production.

### ADR-005: Reformat existing source files with Prettier as part of this pass

**Context:** Configuring Prettier surfaced that most existing files (tab-indented `.astro` files vs. space-indented `package.json`/`tsconfig.json`) didn't match a single consistent style.
**Decision:** Ran `prettier --write` across the repo (excluding `global.css`, `.vscode/`, `.claude/`, and generated/lock files) after previewing every diff — all changes were whitespace-only (2-space → tab, added trailing newlines), verified with `diff` before applying.
**Why:** A formatter with no formatted code to prove it works isn't "configured," and whitespace-only changes carry zero behavioral or visual risk. `.vscode/` and `.claude/` were excluded as editor/tool-specific settings, not project source.

### ADR-006: Pin `typescript` to `^5.9` instead of the latest `7.x`

**Context:** `npm install typescript` resolved to `7.0.2` by default, which falls outside `@astrojs/check@0.9.9`'s declared peer range (`^5.0.0 || ^6.0.0`).
**Decision:** Installed `typescript@^5.7` (resolved to `5.9.3`) instead.
**Why:** Avoids a peer-dependency mismatch that could cause `astro check` to misbehave; `astro check` passes cleanly (0 errors/warnings/hints) on this version.

### ADR-007: Content collections share one base schema instead of four independent ones

**Context:** Production hardening pass required real `blog`/`projects`/`resources`/`courses` collections (superseding ADR-001, which deliberately left collections undefined).
**Decision:** One `baseSchema` (`title`, `description`, `publishDate`, `draft`, `featured`, optional `cover`/`coverAlt`) in `src/content.config.ts`, extended per-collection with exactly one distinguishing field (`category`/`format`/`level`; blog needs none). A `.refine()` requires `coverAlt` whenever `cover` is set.
**Why:** This is what `CONTENT_STRATEGY.md` already committed to ("consistent frontmatter... so a shared card component works without special-casing") — it lets `src/lib/content.ts`'s `getFeatured()` and `<Card>` both work identically across all four collections instead of four bespoke implementations.

### ADR-008: `<Card>` takes flattened props, not a collection entry

**Context:** With real content collections, each collection's `CollectionEntry` type differs slightly (`category` vs `format` vs `level`). `kind` (e.g. "مشروع") was previously stored redundantly 3× per data file.
**Decision:** `<Card>` takes `{ kind, title, description, href, meta, image?, imageAlt? }` — plain values, not a collection entry or the old `PreviewItem` object (removed from `src/types/index.ts`). The page assembling a section passes `kind` once and computes `meta` from whichever field is relevant (formatted date for blog, `category`/`format`/`level` for the others).
**Why:** Decouples the generic, reusable `<Card>` from any one data shape entirely — it has no knowledge of collections at all. `kind`/`meta`-selection is a presentation decision, not content data.

### ADR-009: `<SEO />` is rendered from `Layout.astro`, not imported per-page

**Context:** Task required "every page must use `<SEO />`."
**Decision:** `<SEO />` is rendered once inside `Layout.astro`'s `<head>`, receiving `title`/`description`/`image`/`imageAlt`/`noindex`/`jsonLd` forwarded from `Layout`'s own props. No page imports `<SEO />` directly.
**Why:** A rule that depends on every future page author remembering to import a component is weaker than an architecture where it's structurally impossible to render a page without it. Since every page already must use `Layout` (or `PageLayout`, which itself wraps `Layout`), every page gets `<SEO />` for free and can't opt out by omission.

### ADR-010: Image optimization wired but not exercised by real content

**Context:** Task asked to "prepare the project for future author images and project thumbnails" while also requiring zero visual regressions.
**Decision:** `<Card>`/`<Hero>` accept optional `image: ImageMetadata`/`imageAlt` props rendering `astro:assets`'s `<Image>`; the content schema's `cover`/`coverAlt` fields feed this. No current content entry sets `cover`, and the homepage passes no author photo — so today's output is text-only, identical to before this pass.
**Why:** Building the capability without fabricating placeholder photography satisfies both constraints at once — real assets can be dropped in later with zero component changes, and nothing about the current design changes until they are.

### ADR-011: RSS implemented (supersedes ADR-002's deferral)

**Context:** ADR-002 deferred `@astrojs/rss` until the `blog` collection had real content, since a feed page didn't make sense against empty data.
**Decision:** The `blog` collection now has three real, non-draft entries, so `@astrojs/rss` was installed and `src/pages/rss.xml.ts` added, serving all non-draft posts newest-first. Discoverable via `<SEO />`'s `<link rel="alternate">`, excluded from the sitemap via `astro.config.mjs`'s sitemap `filter`.
**Why:** The condition ADR-002 was waiting on is now met; this is that plan executing, not a new decision to revisit.

### ADR-012: New `<Container>` component replaces duplicated width-constraint CSS

**Context:** `.container`/`.header-container`/`.footer-container` previously shared width/margin-inline rules via a grouped CSS selector — correct in CSS terms, but every consuming component still had to know and type the right class name by hand.
**Decision:** `src/components/Container.astro` renders `<div class:list={['container', class]}>`; `Section`, `Hero`, `Header`, `Footer` all use it (passing their own extra layout class, e.g. `class="hero-grid"`) instead of writing `<div class="container ...">` by hand.
**Why:** Moves the "how do I get the standard page-width constraint" answer from "remember the CSS class name" to "use the component" — the more misuse-resistant version of the same rule, and what the task asked for as a named reusable primitive.

### ADR-013: New `PageLayout.astro`, not yet consumed by any page

**Context:** Task asked for a reusable page layout distinct from the base `Layout.astro` shell.
**Decision:** `PageLayout.astro` wraps `Layout` with a standard title/eyebrow/description heading `<Section>`, for future content pages (`/about`, `/blog/[slug]`, listing pages, etc.). The homepage keeps using `Layout` directly since it has a bespoke `<Hero>`, not a generic heading.
**Why:** Zero risk to add now (nothing currently renders it, so zero visual impact) and saves the next page built from re-implementing the same heading pattern from scratch.

### ADR-014: One-off portfolio page for Mohammed Al-Jabri committed as a static file under `public/`, deliberately outside the Astro architecture

**Context:** Requested deliverable was a single self-contained HTML file (embedded CSS/JS, animated gradient background, glassmorphism, Font Awesome 6 via CDN, EN/AR toggle) for an unrelated person/profession (Additive Manufacturing & 3D Printing specialist) with no connection to this site's existing content, nav, or design tokens. Rebuilding it as a proper Astro page would mean a new content collection entry, `global.css` tokens for a second unrelated visual language, and swapping the CDN icon set for inline SVG — none of which fit a single throwaway page and would pollute the design system for content that isn't part of this site.
**Decision:** Committed as `public/mohammed-aljabri-portfolio.html`, served byte-for-byte like `robots.txt`/favicons — not a `src/pages/` route, not in a content collection, no `<Layout>`/`<SEO />`, no design tokens, and not linked from nav. It is not hand-added to the sitemap; if `@astrojs/sitemap` picks it up from the build output that's incidental, not a deliberate inclusion.
**Why:** This is a one-off exception, not a precedent — the constitution's page/content/styling rules assume pages that are part of this site's actual content and design system. Recording it here (per the "explain architectural decisions before large changes" / "never modify unrelated files" rules) rather than silently bending `src/pages/` conventions to fit it.

### ADR-015: Homepage (`/`) rebuilt to pixel-match an externally supplied reference design, with hardcoded colors and two CDN dependencies — scoped to this one page only

**Context:** Project owner supplied a complete standalone reference HTML file (dark animated gradient, glassmorphism, Font Awesome 6 + Google Fonts via CDN, counter/fade-in/underline animations) with an explicit instruction not to redesign, simplify, or reinterpret it, and to preserve it "exactly." The existing homepage (`src/pages/index.astro`) already had the identical content structure (mini-header/nav, intro, Selected Work, About, Contact, mini-footer) under `Layout`'s `hideChrome`/`fullBleed` bypass from a prior redesign, just with the site's light/token-based visual language instead.
**Decision:** Rebuilt `index.astro`'s markup 1:1 against the reference and moved its CSS into the page's scoped `<style>` block verbatim (hardcoded hex/rgba values, not `global.css` tokens), added `<link>` tags for Google Fonts (Inter + Cairo) and Font Awesome 6 via a `<slot name="head" />` added to `Layout.astro` for this purpose, and ported the reference's counter/`IntersectionObserver` fade-in script as a page-local `<script>`. `selectedWork.ts` gained an `icon` field (Font Awesome name) since the reference shows a distinct icon per project. Everything is scoped to this one route — `global.css`, other pages, and the site's design tokens are untouched.
**Why:** The project owner's explicit instruction is a one-off override of the CSS-tokens/no-new-CDN-dependency rules for this specific page, per this file's own opening line ("an explicit instruction from the project owner wins for that change"). The `<slot name="head" />` addition to `Layout.astro` is small and purely additive (an empty slot, zero effect on any page that doesn't fill it) so it doesn't itself need a separate ADR entry — it's called out here only because it's the mechanism that made the CDN links possible without hand-writing a second `<head>` per the "never render `<head>` by hand" rule.

### ADR-016: Homepage given full creative-direction authority — restructured off the single-card layout, Font Awesome CDN dropped for bespoke inline SVG icons, Cairo dropped as unused dead weight

**Context:** Project owner granted full design authority ("Lead Product Designer... you do not need my permission") to bring the homepage to a "Stripe/Linear/Vercel/Figma/Framer/Raycast/Notion"-grade quality bar, explicitly authorizing redesign of layout, typography, animation, and component proportions, while keeping the dark/glassmorphism/animated-background direction from ADR-015. Auditing that version found: (1) the entire page — nav, hero, and every section — was nested inside one giant centered glass card, which reads as a template widget rather than an editorial page; (2) Font Awesome 6's full CSS + two webfont files shipped ~350KB over CDN to render eight glyphs; (3) the Google Fonts request included `Cairo`, which never renders — this page is English-only (`lang="en"`) and `font-family: 'Inter', 'Cairo', sans-serif` only falls through to a fallback for characters Inter can't cover, which never happens for English text; (4) decorative Font-Awesome icons infinitely spun/scaled inside every section heading with no functional purpose; (5) hover states relied on cartoonish icon-rotation gimmicks.
**Decision:** Restructured the page to full-bleed page flow: a sticky nav bar (transparent → glass once scrolled, via a small scroll listener), an unboxed hero (typography sitting directly on the animated gradient for contrast), and glass treatment reserved for specific elevated components (work-item rows, the About stat band, the closing Contact CTA card) rather than one undifferentiated container — variety in glass usage instead of uniform card-in-a-card nesting. Replaced Font Awesome entirely with `src/components/Icon.astro`, a small reusable component rendering six bespoke stroke-only inline SVGs (mail, message, print, shield, factory, droplet); `selectedWork.ts`'s `icon` field now references these names instead of Font Awesome class names. Dropped the unused `Cairo` family from the Google Fonts request and corrected it to request the four weights actually used (400/500/600/700) instead of an under-specified `opsz` axis. Removed the infinite icon-pulse animation from section headings (replaced with a static uppercase "eyebrow" label per section for hierarchy) and toned down hover states to restrained lift/color transitions. Introduced a page-scoped CSS custom-property palette (colors, glass tokens, spacing, easing) on `.page-bg` to eliminate repeated magic values within this page's own stylesheet. All of this remains scoped to `index.astro` + the new `Icon.astro` component — `global.css`, other routes, and the site's token system are untouched.
**Why:** These were real, identifiable weaknesses against the requested quality bar (Stripe/Linear/etc. use full page flow with glass as an accent, not as page-wide chrome; they don't ship an icon-library CDN for a handful of glyphs; they don't animate decoration with no purpose). Removing Font Awesome and the unused Cairo family directly serves the brief's explicit "optimize for Lighthouse" and "remove dead code" directives — this is a straight subtraction of ~350KB+ of unused network weight, not a stylistic swap. **Also fixed in this pass:** a CSS specificity bug in the page's own reset rule (`.page-bg *`) — Astro's scoped-style compiler rewrites a bare `*` into a standalone attribute selector, which unexpectedly made the reset _more_ specific than any single-class component rule and silently zeroed out most padding/margin on the page. Fixed by wrapping the reset in `:where(...)` (zero-specificity by spec), which is now the correct pattern for any future scoped reset in this codebase.

### ADR-017: Selected Work rebuilt as `<ProjectCard>`, adapting a Uiverse reference's motion language to this page's glass palette

**Context:** Project owner supplied a Uiverse card design (white background, lift-on-hover, shine sweep, radial glow, image scale, badge reveal, pulsing circular action button) and asked for its interaction/motion language to replace the Selected Work list, adapted to this page's dark glassmorphism instead of the original white card.
**Decision:** New `src/components/ProjectCard.astro` — a reusable, flattened-props component (`title`, `description`, `status?`, `image?`, `imageAlt?`, `href?`, `category?`, `meta?`, `actionLabel?`, `icon?`) per the "generic display components take flattened props" component rule. It reads the same `--accent`/`--glass-bg`/`--ease-premium`/etc. custom properties already defined on `.page-bg` (inherited via normal CSS cascade, unaffected by Astro's style scoping), with hardcoded fallbacks via `var(--x, fallback)` so the component still renders sensibly if ever reused somewhere those tokens aren't defined. None of the three real projects have a destination URL, so `href` is omitted for all of them; the component renders as a plain non-interactive `<article>` in that case rather than fabricating a dead link — the circular action icon is always `aria-hidden` and purely decorative (never independently focusable), since nesting a real link/button inside the whole-card `<a>` that appears when `href` is supplied would be invalid HTML. `selectedWork.ts`'s `label` field was renamed to `category` (it was already unused in rendering) and given real category text per project. The former `.work-list` flex/grid CSS was deleted and replaced with a `.project-grid` (3 columns → 2 at 900px → 1 at 640px, matching this page's existing breakpoint rhythm) in `index.astro`; `Icon.astro` gained an `arrow` glyph for the action button, keeping the bespoke icon system from ADR-016 as the single source instead of hand-inlining a one-off SVG.
**Why:** Keeps the card generic/reusable (flattened props, not a `SelectedWorkItem` object) consistent with `ADR-008`'s rationale for `<Card>`. Declining to fabricate `href`s for projects with no real URL avoids a dead-link/fake-affordance antipattern while still fully delivering the requested visual/motion language — the button's pulse and the card's shine/glow/lift all still play on hover and on keyboard focus (`:focus-visible`) whether or not the card happens to be a link. All motion is disabled under `prefers-reduced-motion`, per this page's established pattern.

### ADR-018: `<LanguageSwitcher>` — Arabic points at `/about`, not a fabricated Arabic translation of the homepage

**Context:** Project owner asked for a premium EN/AR language switcher on the homepage, inspired by a Uiverse button's motion but redesigned for this site, architected so more languages (and eventually `/en/`/`/ar/`-prefixed routing) can be added without refactoring. The homepage (`/`) is English/LTR by explicit one-off exception (ADR-014/015); every other route (`/about`, `/blog`, `/projects`) is Arabic/RTL, the site's actual default per `Layout.astro`. Critically, `/` has no Arabic translation of its own copy today — none of the hero bio, Selected Work descriptions, stats labels, or contact copy exist in Arabic. `/about`, however, already carries the real Arabic equivalent of this same personal introduction (same person, same six-years/32-projects/63-workshops/1,340-beneficiaries figures), just under a different page structure.
**Decision:** `src/data/languages.ts` defines each language as `{ code, label, name, href, dir }` — an ordinary array a future language is added to, not a routing scheme baked into the component. `src/components/LanguageSwitcher.astro` renders that list as a segmented glass pill and determines the _active_ entry itself, by matching `Astro.url.pathname` against each `href` — the consuming page passes no props and owns no language state, satisfying "keep the logic isolated." Today `en` → `/` and `ar` → `/about`; clicking "العربية" from the homepage is a real navigation to real, already-correct (`lang="ar" dir="rtl"`) Arabic content, not a same-page toggle that would leave English prose sitting inside a `dir="rtl"` shell. No JavaScript: every option is a plain `<a>`, so it's keyboard-native and needs no script to work; hover/press feedback is pure CSS using the page's established `--ease-premium` easing, and the component defines its own themed `:focus-visible` ring rather than depending on `index.astro`'s scoped rule, which (per ADR-017's discovery about Astro's per-component style scoping) would not actually reach an element rendered by a _child_ component.
**Why:** "Load Arabic content when available, gracefully fall back to English otherwise" is a per-page/per-string policy; applied at the whole-page level, the honest fallback for a page with zero Arabic translation is to send the visitor to the site's real Arabic content, not to mislabel the English page as Arabic. Translating the homepage's copy is a real content decision outside this task's scope (building the switcher), and fabricating placeholder Arabic strings to make an in-place toggle "work" would be worse than pointing at genuine equivalent content that already exists. The config-driven `SiteLanguage[]` list is what makes the architecture future-proof: switching `href` values to `/en/`/`/ar/` later, or adding a third language, touches only `languages.ts`.

### ADR-019: `/ar` — the homepage's real Arabic counterpart, on a shared `<PersonalPortfolio>` template

**Context:** ADR-018 pointed the switcher's Arabic option at `/about` because that was the only real Arabic content available at the time. The project owner then asked for the English and Arabic homepages to reach the same production-quality standard — identical colors, glass, shadows, spacing, typography hierarchy, animation, and component quality, "not a weaker, simpler, or differently colored Arabic version" — which `/about` (the sitewide light Apple-style theme, a different page structure entirely) could never satisfy no matter how much it was polished, because it isn't a translation of the homepage, it's a different page.
**Decision:** Extracted the homepage's markup, styles, and behavior out of `index.astro` into `src/components/PersonalPortfolio.astro` — one template, driven entirely by props (`nav`/`hero`/`work`/`about`/`contact`/`footer`/`email`/`whatsapp`) plus a `dir: 'ltr' | 'rtl'` flag, with one named slot (`about-paragraph`) for the single line whose emphasis placement differs by language. `src/pages/index.astro` and the new `src/pages/ar/index.astro` are now both thin wrappers: each owns only its `Layout` wiring (title/description/jsonLd/htmlLang/htmlDir/locale, plus reciprocal `hreflang` alternate links) and its own copy, rendering the same `<PersonalPortfolio>`. `languages.ts`'s `ar` entry now points at `/ar`. `selectedWork.ts` was restructured so each project carries `{ icon, en: {...}, ar: {...} }` together — one entry per real project, both languages reviewable side by side, instead of two arrays that could drift apart.
RTL correctness came almost for free: the component already used logical properties (`inset-inline-*`, `margin-inline`, flexbox/grid's direction-aware `justify-content`/track order) everywhere except one `left: 0` on the nav-underline, fixed to `inset-inline-start`. Three things needed real, deliberate handling, not just a mirror: (1) **font** — Arabic uses this site's existing self-hosted "Thmanyah Sans" (already the default body font on every other Arabic page) instead of loading a second webfont for one page; (2) **letter-spacing** — Arabic is a connected script where tracking breaks glyph connections, so a `:dir(rtl)` block resets every heading/label's letter-spacing to `normal`, English keeps its original values; (3) **the action-button arrow** on `<ProjectCard>` — pointed up-right regardless of direction, which reads as "backward" once the button sits mirrored on a card's left edge in RTL, so it now flips (`scaleX(-1)`) and its hover-nudge reorders to match, using the same numeric values rather than a second set. QA also caught the phone number (`+968 91899818`) reversing digit groups under the bidi algorithm when placed in RTL text with no surrounding Arabic to anchor it — a well-known class of bug — fixed by isolating it in `<bdi dir="ltr">`.
Arabic copy (hero bio, project descriptions, about paragraph, stats labels, contact CTA) is original writing for this page's structure, not a literal translation of the English sentences — it reuses vocabulary already established on `/about` (e.g. "التصنيع الإضافي" for additive manufacturing) for consistency, keeps organization proper nouns as their standard Arabic name (وزارة الدفاع) or a conventional transliteration (جيندال ستيل عُمان) or unchanged acronym (OQGN) as appropriate, and does not add, round, or otherwise alter any figure (6 years, 32+ projects, 63 workshops, 1,340+ beneficiaries) from the English source.
**Not changed in this pass:** the global Arabic `<Header>`'s home link (`src/components/Header.astro`, via `src/data/nav.ts`) still points at `/`, which is now unambiguously the English page — an Arabic reader clicking "الرئيسية" from `/blog` or `/projects` still lands in English. Repointing it to `/ar` would be the coherent next step, but `Header.astro` is shared sitewide chrome outside this task's scope (the homepage's bilingual pair), so it's recorded here rather than changed inline.
**Why:** "Shared components, not two websites" and "identical production quality" are only achievable if both languages render from the same template — a hand-maintained second copy of ~550 lines of CSS would drift the moment either page changed. Keeping `/about` untouched and separate (rather than repurposing it) avoids breaking whatever already links to it, and keeps `/ar` unambiguously "the homepage in Arabic" rather than conflating two different pages' purposes.

### ADR-020: Homepage copy rewritten from the project owner's CV, adding a new Experience section

**Context:** Project owner asked for every piece of homepage copy (EN + AR) to be rewritten using their CV as the source of truth — not translated literally, not turned into an online résumé, no invented achievements/clients/numbers. The CV (`Mohammed_Aljabri_CV_2026.docx`, confirmed as the authoritative file after multiple differently-named CVs were found in Downloads) named two real employers with dates (Plamore Manufacture, Dec 2024–Jan 2026; 3D Factory, Sep 2020–Nov 2024) and explicitly listed "Experience" as a section to rewrite alongside Hero/About/Selected Work/Contact/Footer — content the homepage didn't have a section for at all.
**Decision:** Added a new `experience` prop and `<section id="experience">` to `PersonalPortfolio.astro` — two compact glass rows (role — company, dates, one-line description each), not a bulleted résumé timeline, styled consistently with the existing card language and picked up automatically by the already-generic `.reveal`/`IntersectionObserver` logic (no JS changes needed). Added a matching "Experience"/"الخبرة" nav entry on both pages. Rewrote hero intro, About paragraph, Selected Work descriptions, Contact copy, and stat labels on both `index.astro` and `ar/index.astro` using specifics the CV actually states (6+ years, FDM production, root-cause diagnostics, print-workflow optimization, contribution to developing and commercializing a locally-built 3D printer, 63 workshops reaching 1,300+ participants across every governorate in Oman) — none of it copied from the CV's own sentences. Relabeled the "Beneficiaries"/"مستفيدون" stat to "Participants"/"مشاركون" to match the CV's own term, and the years-of-experience stat to "6+" (the CV says "6+ years," not exactly six).
The CV lists four additional organizations beyond the three already on Selected Work (Civil Aviation Authority, PDO, SLB, Sultan Haitham City) under "supported additive manufacturing projects for" — these were deliberately **not** added as new project cards, per the brief's own "mention notable organizations only where appropriate; less is better; do not overload the homepage." The existing three (Ministry of Defence, Jindal Steel Oman, OQGN) are all independently confirmed by the CV, so they were kept and their descriptions refreshed rather than replaced.
**Known discrepancy, left as-is:** the "32+ Projects" stat (present on the homepage before this task) does not appear anywhere in the CV text. It wasn't contradicted either — the CV's project list is a set of _named clients_, not a total project count, so there's no basis to conclude 32 is wrong. Per "do not fabricate numbers," removing a previously-established figure on no positive evidence would be as unjustified as inventing one, so it was left unchanged. Flagged here for the project owner to confirm independently.
A real bug surfaced during QA: Prettier's line-wrap of a multi-line `<strong>…</strong>` tag in the Arabic about-paragraph placed the closing `>` on its own line directly before the next word, and Astro rendered that boundary with no whitespace, producing "63 ورشة تدريبيةاستفاد" (two words fused together). Fixed with explicit `{' '}` expressions, which are immune to this class of whitespace-collapsing regardless of how a formatter wraps the surrounding markup.
**Why:** A dedicated Experience section is the only way to surface real, named-employer career history — the single strongest credibility signal in the CV — without cramming it into the About paragraph or turning Selected Work into a résumé. Keeping the entry count in Selected Work unchanged (rather than expanding to 6-7 cards for every CV-named client) directly follows the brief's explicit restraint instruction; the same organizations remain available to mention narratively later if ever needed.
